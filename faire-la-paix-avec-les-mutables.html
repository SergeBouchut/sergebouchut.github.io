Faire la paix avec les mutables

Sun 17 June 2018

En python, certaines structures de données sont dites "mutables: : on peut en modifier les valeurs, après les avoir instanciées : c’est naturellement le cas des objets, des dictionnaires, des listes (mais pas des tuples), et bien d’autres.

Les mutables sont nos amis

Tentons de modifier le contenu d’une liste, en conservant sont identité (sans en instancier une nouvelle).

xmens = ['tornade', 'cyclope']

id(xmens)
140229166148488

xmens.append('phoenix')
['tornade', 'cyclope', 'phoenix']

id(xmens)
140229166148488  # same id

Rien de surprenant, il semble assez naturel, avec un langage haut niveau comme Python, de pouvoir librement ajouter des élément à une liste. Si les listes n'étaient pas "mutables", on devrait en instancier une nouvelle à chaque fois.

xmens = ['tornade', 'cyclope']

id(xmens)
140229157348616

xmens = [*xmens, 'malicia']
['tornade', 'cyclope', 'malicia']

id(xmens)
140229204490952  # other id

Au passage, on note la distinction entre égalité (des valeurs) et identité (des instances).

['tornade', 'cyclope'] == ['tornade', 'cyclope']
True  # same values

['tornade', 'cyclope'] is ['tornade', 'cyclope']
False  # not the same instances

Parfois, cela offre des fonctionnalités assez pratiques. Par exemple, nul besoin de retourner une liste modifiée dans une fonction.

def hire(mutant, team):
    team.append(mutant)

    xmens = ['tornade', 'cyclope']
    hire('wolverine', xmens)

print(xmens)
['tornade', 'cyclope', 'wolverine']

Des amis qui n’en font qu’à leur tête !

Mais, cela peut aussi amener à des comportements surprenants, si on définis un mutable comme valeur par défaut d'un argument. Dans l’exemple, on créer et retourne une nouvelle liste si aucune n’est fournie.

def hire(mutant, team=[]):
"""Create a new team, if no team provided."""
    team.append(mutant)
    return team

    xmens = hire('charles-xavier')
    ['charles-xavier']

    villains = hire('magneto')
    ['charles-xavier', 'magneto']  # ?!

Non, non, il semblerait que ce ne soit pas un anomalie, mais bien un "choix d'implémentation".

En Python, les fonctions (tout comme les classes) sont des objets de première classe, on peut les assigner, accéder aux propriétés, etc.

Voir: https://fr.wikipedia.org/wiki/Objet_de_premi%C3%A8re_classe

En pratique, ça donne :

x = hire

x.__class__
function

x.__name__
hire

team = []
x('wolverine', team)

print(team)
['worlverine']

Les valeurs par défauts des arguments font aussi partis des propriétés de la fonction. Revenons sur l’exemple de tout à l’heure.

def hire(mutant, team=[]):
"""Create a new team, if no team provided."""
    team.append(mutant)
    return team

hire.__defaults__
([],)

hire('charles-xavier')
hire.__defaults__
(['charles-xavier'],)

hire('magneto')
hire.__defaults__
(['charles-xavier', 'magneto'],)

La valeur par défaut étant une propriété mutable, elle est susceptible d’évoluer à chaque appel de la fonction.

Faut reconnaître que c’est un peu traître et à priori pas considéré comme une bonne pratique. Pylint, par exemple retournera un message d’avertissement.

W0102: Dangerous default value [] as argument.

Voir: http://pylint-messages.wikidot.com/messages:w0102

Un remède contre les mutations

On peut, par exemple, définir None comme valeur par défaut, puis créer la liste à l’intérieur de la fonction. Ainsi, la liste ne sera pas liée aux propriétés de la fonction.

def hire(mutant, team=None):
    """Create a new team, if no team provided."""
    if team is None:
         team = []
    team.append(mutant)
    return team

hire.__defaults__
(None,)

hire('charles-xavier')
hire.__defaults__
(None,)

hire('magneto')
hire.__defaults__
(None,)

Notons, qu’on rencontre le même problématique avec les valeurs par défaut des attributs de classe, qui pourront être modifiés entre les instances.

class Team:
    mutants = []
    def hire(self, mutant):
        self.mutants.append(mutant)

xmens = Team()
xmens.hire('charles-xavier')
xmens.mutants == ['charles-xavier']

villains = Team()
villains.hire('magneto')
villains.mutants == ['charles-xavier', 'magneto']

Team.mutants == ['charles-xavier', 'magneto']

Et le remède est du même acabit que pour les arguments des fonctions.

class Team:
    mutants = None
    def hire(self, mutant):
        if self.mutant is None:
            self.mutant = []
        self.mutants.append(mutant)

xmens = Team()
xmens.hire('charles-xavier')
xmens.mutants == ['charles-xavier']

villains = Team()
villains.hire('magneto')
villains.mutants == ['magneto']

Team.mutants == None

Personnellement, même si je comprends les raisons techniques d'un tel comportement, je ne suis pas convaincu de son intérêt sur un plan pratique.

Il y a bien d’autre manière, plus simples et plus explicites, de rendre des données persistantes entre des objets ou des appels de fonctions. Celle-ci est contre-intuitive pour un langage haut niveau comme Python, où le développeur n'a pas à se préoccuper des spécificités sous jacentes à l'implémentation.

J'ai envie d'appuyer mon point de vue avec quelques dogmes officieux de Python.

  • Explicit is better than implicit.
  • Simple is better than complex.
  • Practicality beats purity.
  • There should be one — and preferably only one — obvious way to do it.

J'imagine, qu'il existe une bonne raison d'avoir conserver un tel comportement mais je n'en ai pas trouvé d'explication.

Note : les mutables ne seront pas acceptés en valeur par défaut des attributs des Data Classes, qui arrivent avec Python 3.7. - https://www.python.org/dev/peps/pep-0557/#mutable-default-values - https://github.com/ericvsmith/dataclasses/issues/3
